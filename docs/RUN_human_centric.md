# Running the style-matched human-centric control set

This is the end-to-end sequence for experiment (b): a 3rd-person human-situation
stimulus set generated by the *same* pipeline as the AI-centric set (same Shaver
concepts, same noise injection, same length target, same emotion-word
prohibition), differing only in content/person/addressee. It is the clean
control for the structural confound behind the asymmetric-generalization result.

## 0. One-time setup

```bash
uv venv --python 3.11 .venv
source .venv/bin/activate
uv pip install -r requirements-vllm022.txt   # vLLM 0.22.1 + torch 2.11/CUDA13 + analysis deps
cp .env.example .env                          # then fill in OPENROUTER_API_KEY and OPENAI_API_KEY
```

Activation extraction now uses **runtime forward hooks** registered via
`llm.collective_rpc` (see `src/llm/activation_hooks.py`) — there is no longer any
need to overwrite vLLM's model source. The old `vllm_changes/` files,
`scripts/apply_vllm_patches.py`, and the 0.10.0 pin in `requirements.txt` are
**superseded** and can be removed. (`.venv-vllm010` is kept only as a fallback.)

Target model weights live under `/home/models/`. Available: Qwen2.5-3B/7B/14B/32B,
Qwen3-0.6B/1.7B/4B, Llama-3.1-8B, Llama-3.2-3B. **Llama-2-7b-chat-hf (the paper's
model) is not present** — download it, or use one of the above.

## 1. Generate the human-centric stimuli (OpenRouter, ~$30–40)

`src/llm/generate_prompts.py` is already set to `PROMPT_STYLE = "human_centric"`.
It writes to `data/01_stimuli/generated_human_prompts/generated_human_emotional_prompts_batched.csv`.

```bash
python src/llm/generate_prompts.py
```

Sanity-check the output before spending GPU time:
- prompts are 3rd person (no "you"/"I"), no AI/tech/existential references;
- no occurrences of the literal target emotion word;
- length/structure visually comparable to the AI-centric set.

## 2. Inference + activation extraction (local GPUs)

Generation is **batched** (vLLM continuous batching), and activations are
captured by forward hooks. Last-token only is the default (`[1, hidden]` per
layer — ~25× less disk than full-sequence, no loss for last-token probing/RSA).
`max_tokens=256` / `temperature=1.0` match the existing AI-set runs.

```bash
# one model per GPU (run both concurrently on different GPUs)
CUDA_VISIBLE_DEVICES=0 python -m src.llm.generate_with_hooks \
    --model /home/models/Qwen2.5-14B-Instruct --dataset generated_human_prompts
CUDA_VISIBLE_DEVICES=1 python -m src.llm.generate_with_hooks \
    --model /home/models/Llama-2-7b-chat-hf   --dataset generated_human_prompts

# a model too big for one card: tensor-parallel across GPUs
CUDA_VISIBLE_DEVICES=0,1,2,3 python -m src.llm.generate_with_hooks \
    --model /home/models/Qwen2.5-32B-Instruct --tp 4 --dataset generated_human_prompts
```

Useful flags: `--tp/--pp` (vLLM tensor/pipeline parallel, for big models),
`--chunk-size`, `--max-tokens`, `--temperature`, `--resume`, `--testing` (first
10 prompts), and `--poolings` (comma-separated subset of
`last_token,mean,max,min,amp`; default `last_token`, which is what the probes
use). `VLLM_ALLOW_INSECURE_SERIALIZATION=1` is set automatically (needed so the
hook closures reach the workers).

Activations are pooled at capture time and written as a few **`chunk_*.pkl`**
files per run (not millions of per-tensor `.pt` files), under
`data/03_activations/activations_<run>/`. Storage ≈ #prompts × #layers × hidden
× 4 B per pooling (~20 GB for Qwen-14B last-token over 20k prompts).

To fan **one** run across all 8 GPUs, use the data-parallel launcher
(`scripts/run_dp.py`) — it spawns one process per GPU group over disjoint prompt
slices, shares one activations dir, and merges the per-shard outputs:

```bash
# 8-way data parallel (one model per GPU)
python scripts/run_dp.py --model /home/models/Qwen2.5-14B-Instruct \
    --dataset generated_human_prompts --gpus 0,1,2,3,4,5,6,7
# 4 shards x 2-GPU tensor parallel, for a model too big for one card
python scripts/run_dp.py --model /home/models/Qwen2.5-32B-Instruct \
    --gpus 0,1,2,3,4,5,6,7 --tp 2
```

It prints the run id to use in step 3, and supports `--resume --run-id <id>` if a
shard fails. (In-process `data_parallel_size>1` is not supported by vLLM's
offline API; `run_dp.py` is the multi-process equivalent.)

The run prints a run id like `Qwen2.5-14B-Instruct_YYYYMMDD_HHMMSS`.

## 3. Annotate responses (GPT-5-mini Batch API, ~$35–100 per model)

In `src/nlp/responses_annotation.py` set `RUN_TO_PROCESS` to the run id from
step 2 (`DATASET` is already `generated_human_prompts`; `BATCH_SIZE=500`).

```bash
python src/nlp/responses_annotation.py
```

## 4. Consolidate, merge, probe

`output_and_activation_pairing.py` now just consolidates the chunk pkls + the
outputs JSONL into the nested-DataFrame pkl the downstream expects (set
`RUN_TO_LOAD`/`DATASET_USED` at the top of it first). The per-chunk pkls can be
deleted afterwards to reclaim space.

```bash
python scripts/output_and_activation_pairing.py        # -> <dataset>_<run>.pkl
python scripts/annotations_and_activations_merge.py
python src/probes/train_linear_probes_on_annotations.py
python src/probes/interdataset_cross_test.py   # AI <-> human-centric cross-domain test
```

The cross-test now contrasts AI-centric vs the **style-matched** human-centric
set, isolating content from generation-style/length/person.
